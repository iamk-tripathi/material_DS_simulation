# Text Field

Text Field captures user-entered text with labeled input variants and state feedback. It solves the need to collect form input while keeping labels, helper text, and validation states readable.

## Variants

<DocSection title="Filled" description="Use inside surfaces where the input needs a stronger background treatment.">
  <ComponentPreview caption="Filled Text Field" note="Use the filled variant inside grouped surfaces or denser form areas.">
    <ThemeProvider theme={theme}>
      <DSTextField variant="filled" label="Label" helperText="Helper text" />
    </ThemeProvider>
  </ComponentPreview>
</DocSection>

<DocSection title="Outlined" description="Use on plain backgrounds where the field boundary should remain clearly visible.">
  <ComponentPreview caption="Outlined Text Field" note="Use the outlined variant when the surrounding surface is already visually quiet.">
    <ThemeProvider theme={theme}>
      <DSTextField variant="outlined" label="Label" helperText="Helper text" />
    </ThemeProvider>
  </ComponentPreview>
</DocSection>

## States

<DocSection title="Error" description="Use when validation fails and the user needs correction guidance.">

## Do's and Don'ts

<DosDonts
  dos={[
    'Always provide a visible label',
    'Use helper text to clarify input expectations or validation',
    'Match filled and outlined variants to the surrounding surface',
    'Use error state only when validation fails',
    'Keep field labels and helper text concise',
  ]}
  donts={[
    'Use placeholder text as the only label',
    'Disable fields without explaining availability through context',
    'Mix filled and outlined fields randomly in the same form section',
    'Use error styling without helpful guidance',
    'Rely on color alone to communicate field state',
  ]}
/>

## Accessibility

<DocSection title="Keyboard Navigation">
  Users should be able to tab into the field, type, and move between fields without losing visible focus.
</DocSection>

<DocSection title="Screen Reader">
  Text fields should keep a visible label and helper text relationship so the input purpose remains clear.
</DocSection>

<DocSection title="Contrast">
  Error and focus treatments should remain distinguishable, and validation should not rely on color alone.
</DocSection>
